Day 9 · Session 2

What is Conditional Logic?

Conditional logic is the difference between an automation that does the same thing every time and one that thinks before it acts.

In plain language: If X is true, do A. If Y is true, do B. If nothing matches, do C (the fallback).

You already use this logic every day without calling it that:

  • If the client is urgent, call them. If not, send an email.
  • If the invoice is unpaid after 7 days, send a reminder. If paid, send a thank-you.
  • If the post is for Instagram, add hashtags. If it is for LinkedIn, keep the tone professional.

In Make.com, this is handled with a module called a Router. You set a filter condition on each route, and the automation directs itself to the right action depending on what it finds.

The 5 Key Terms

TermWhat it meansIn Make.com
TriggerThe event that starts the automationNew row in Google Sheets
ConditionThe rule being testedPlatform = "Instagram"?
PathThe branch taken when a condition is trueMake.com Router route
FallbackWhat runs when NO condition matchesThe final route with no filter
LogA written record of every outcomeAppend row to a Sheets log tab
Why the fallback matters

Most beginners skip the fallback. If someone adds a row with Platform = "TikTok" and there is no fallback route, the Scenario fails silently — no error, no alert, no post scheduled. Nobody knows. The fallback is your safety net. It catches anything unexpected and notifies a human to review it.

Why the log matters

The log is your paper trail. When something breaks (and it will), you need to know: when did it trigger, what was the platform, what happened? Without a log, debugging is pure guesswork. With one, you pull up the sheet and the answer is right there.

Case sensitivity — the most common failure point

"Instagram" and "instagram" are not the same string in Make.com filters. Your team must enter platform names exactly as written. To avoid this, use the text function {{lower(Platform)}} in your filter and match against lowercase values like "instagram", "twitter", "linkedin." This way it works regardless of how the team typed it in the sheet.

Day 9 · Week 1 Review

Key Takeaways — Week 1

Days 1–5. These are the concepts you will be assessed on. Know them cold.

Day 1 — AI Foundations
  • LLMs predict tokens, not think. No memory, beliefs, or understanding — they pattern-match on training data. Every output needs your judgment before it reaches a client.
  • Context window = the maximum text the AI can see at once. Past the limit, earlier content is silently dropped. Longer conversations need active context management.
  • Zero-Trust Policy: on free tiers, assume your inputs can train future models. Never paste client data, credentials, or confidential information into a free AI tool.
  • Hallucination = AI confidently stating something false. It doesn’t know it’s wrong — it predicted a plausible token. Always verify facts, numbers, names, and links before forwarding.
  • Free vs paid tiers differ in privacy: free tiers often allow training on your inputs. Pro/paid tiers typically opt you out. Know your plan before pasting anything sensitive.
  • Metaprompting = asking the AI to write or improve the prompt for you. Useful when you know what you want but not how to phrase it.
  • Negative constraints tell the AI what NOT to do — often more powerful than positive instructions. (“Do not use bullet points.” “Do not mention pricing.”)
  • Benchmarking: test the Big Four (Claude, ChatGPT, Gemini, Grok) against the same prompt for high-stakes tasks. No single model wins every time.
  • Iteration loop: prompt → output → evaluate → refine → repeat. Bad outputs are not failure — they are input for the next prompt. Expect 3–5 iterations for anything important.
Day 2 — Prompt Blocks & Right Tool
  • Six prompt blocks: Role, Task, Context, Constraints, Examples, Format. Each has a job. Missing one is usually why the output is wrong.
  • Role = who the AI is; Task = the deliverable; Context = background it needs; Constraints = what to avoid; Examples = show, don’t just tell; Format = length, structure, tone.
  • Format matters more than most people expect: “bullet points, max 3 per section, plain language” produces something usable. “Be professional” produces nothing actionable.
  • Four failure modes: too vague (no task/context), too broad (no constraints), wrong format (output unusable as-is), no examples (AI guesses your style).
  • Debug loop: identify the failure mode first. Change ONE block. Re-run. Never rewrite the whole prompt — you lose the ability to know what fixed it.
  • Zero-shot vs few-shot: 0 examples = zero-shot. 1 example = one-shot. 2+ = few-shot. More examples = better style and format matching for recurring content tasks.
  • Prompting vs context engineering: prompting = what you type in one message. Context engineering = what you set up before the conversation (project instructions, reference docs, stored examples).
  • Right tool for the job: Chat = quick one-off answers. Automation = repeatable trigger+action tasks. Agent = multi-step work needing tool access.
Day 3 — Claude Deep Dive
  • Artifacts = finished, reusable outputs in a side panel: documents, code, tables, HTML. A deliverable you can copy, edit, or ship — not a conversation.
  • MCP (Model Context Protocol) = a universal translator that lets Claude connect to external tools (Google Drive, GitHub, calendars) without custom code for each integration.
  • CLI (Command Line Interface) = text-based terminal interface. Claude Code, scripts, and agents run here — outside the browser, directly on your machine.
  • Projects give Claude persistent context across conversations — system instructions, files, and notes stay loaded every session. Skills give it domain-specific instructions for recurring tasks.
  • Claude Code vs Cowork: Claude Code = terminal CLI for coding tasks on your machine. Cowork = the Projects panel inside Claude.ai — for ongoing non-code work with file and conversation continuity.
  • Fresh chat vs Project: a fresh chat has no memory of previous conversations. A Project retains context across sessions. Use Projects for any ongoing client work where continuity matters.
  • When to use which model: Claude for nuanced writing, analysis, reasoning. ChatGPT for broad general use. Gemini for Google Workspace integration. Grok for real-time Twitter/X context.
  • Context engineering = structuring everything you feed the AI so it has exactly what it needs and nothing it doesn’t. The habit that separates power users from casual ones.
Day 9 · Week 1 Review (cont.)

Key Takeaways — Days 4 & 5

Day 4 — Automation Foundations
  • Every automation is one shape: trigger → (filter) → action(s) → result. This never changes regardless of tool.
  • Four automation types: simple (1 trigger → 1 action), scheduled (time-based), multi-step (chain of actions), conditional (if/else branching).
  • The trigger = the event that starts it. Filter = decides whether to run. Action = what it does. Result = the outcome (post published, email sent, row updated).
  • Four-question map before building: What triggers it? What does it do? What are the edge cases? What can go wrong?
  • Human-in-the-loop rule: anything that sends a message to a client, makes a permanent change, or handles money needs a human checkpoint before it fires.
  • Data compliance: never automate sending client PII (name + email + phone together) to a third-party tool without checking its privacy policy and getting client consent first.
  • Debugging ladder: first check if the trigger fired (run history). Then check if the filter matched. Then check if the action had the right data. Always work downstream from the trigger — never start from the output.
  • QA before going live: test with perfect data, messy data, empty fields, and unexpected values. Test the fallback — not just the happy path.
Day 5 — Building Automations
  • Build the trigger alone first. Test it. Confirm it fires. Only then add the first action.
  • Add one module at a time. Test after each. Complexity multiplies problems — especially untested ones.
  • Your prompt IS the logic inside an AI step. The automation output quality equals the prompt quality. Garbage in, garbage out.
  • Label every module with a descriptive name. “Google Sheets 3” tells you nothing. “Update Status Column” tells you everything when debugging at midnight.
  • Never connect to live client data on the first run. Use a test spreadsheet or test contact until every path is verified.
  • QA checklist: perfect data? Messy data? Missing fields? Edge cases? Never go live without testing all four.
  • Simplest working version first. A 3-module Scenario that runs reliably beats a 12-module one you half-understand.
  • Debugging: disable all modules except one. Re-enable one at a time until the break appears. Isolate variables, then fix.
The Week 1 thread

Week 1 is the same principle at different layers: structured inputs produce reliable outputs. A prompt with six blocks. An automation with a trigger, filter, action, and fallback. A Claude Project with proper context. All the same idea — applied to text, logic, and systems. Week 2 applies it inside real client tools.

Day 9 · Week 2 Review

Key Takeaways — Week 2

Days 6–9. These concepts carry directly into client work.

Day 6 — GoHighLevel Foundations
  • GHL replaces 8–10 tools: CRM, email, SMS, social planner, calendar, funnels, reputation management, conversations inbox — one platform.
  • Agency account = the parent. Each client gets a sub-account — a completely separate workspace. Never mix client data across sub-accounts.
  • Contacts are the central record. Every message, pipeline stage, tag, and automation connects back to a contact.
  • Tags = simple labels. They trigger workflows, create smart lists, and segment messaging. One tag can start an entire automation sequence.
  • Custom fields store specific data per contact. Smart lists update dynamically based on conditions — never manually sorted.
  • Pipelines track where each lead sits. Moving a contact to a new stage can trigger a workflow automatically.
  • Conversations inbox = unified messaging: SMS, email, Facebook Messenger, Instagram DMs, and WhatsApp all in one view per contact — full cross-channel history in one place.
  • Reputation management: GHL automates review requests — sends an SMS or email 24 hours after job completion asking for a Google or Facebook review. High-volume clients use this to build review counts systematically.
Day 7 — GHL Workflows & API
  • Workflow building blocks: trigger (what starts it) + condition (if/else) + action (what it does). Same shape as every automation from Week 1.
  • 7 essential recipes: speed-to-lead, appointment reminder, no-show follow-up, long-term nurture, review request, missed-call text-back, database reactivation.
  • Speed-to-lead timing: responding within 5 minutes is 9× more likely to convert than 30 minutes. The recipe sends an instant SMS the moment a form is submitted — no manual action needed.
  • Missed-call text-back = highest-ROI recipe. Someone calls, gets no answer → instant SMS: “Hi, missed your call! When’s a good time?” Clients routinely recover leads they would have lost permanently.
  • Bulk vs workflow: bulk = you manually select contacts and push an action now (one-time batch). Workflow = automated trigger, runs forever. Never confuse them.
  • API = waiter (you ask, it fetches). Webhook = doorbell (external system pushes to you instantly). Different use cases — different trigger types.
  • 3 levels of AI in GHL: draft assist (suggestions in conversations), Conversation AI (live AI replies to leads), workflow AI (AI step mid-automation for content or decisions).
  • API keys = passwords. Store in a password manager. Never share in Docs, Slack, or email. If exposed, rotate immediately.
Day 8 — AI Meetings & PM
  • AI note-takers join calls, transcribe in real time, and generate summaries with decisions and action items automatically.
  • Meeting-to-action pipeline: meeting → note-taker records → Claude processes transcript → VA verifies output → tasks routed to PM tool.
  • VA role = the judgment layer. AI captures and drafts. You verify accuracy, catch hallucinations, route tasks. Never forward raw AI output to a client without review.
  • Three verification checks: (1) decisions attributed to the right person? (2) action items realistic given project context? (3) summary missing important context the AI didn’t catch?
  • Fathom = free unlimited, works best on Zoom, generates structured summaries with action items. Best entry-level choice for VA work on Zoom-based clients.
  • PM tools (ClickUp, Asana, Trello, Notion): every task needs an owner, a deadline, and enough context to action without follow-up questions.
  • KeyCommand: Starter/Pro/Agency tiers. Genes = AI-generated content from an executive’s existing posts. Approval workflow = content drafted, holds for human sign-off before posting. Critical for client-facing accounts.
Day 9 — Conditional Logic
  • Router splits one flow into multiple paths. Only one path runs per trigger — the first condition that matches wins. Route order matters.
  • Every path needs three things: the condition check, the action(s) for that path, and a log entry recording the result.
  • Fallback path is non-negotiable. Without it, unexpected inputs fail silently — you find out from the client, not from a log.
  • Case sensitivity kills automations. “Instagram” and “instagram” are different strings. Always use lower() to normalise text before matching in any filter.
  • Operations math: each module execution = 1 op. A 5-module Scenario × 200 rows/month = 1,000 ops = the entire free tier. Plan volume before going live.
  • When no route matches and there’s no fallback: Make.com stops the run with an error, not a graceful skip. Silent errors in production = missing posts and confused clients.
  • lower() syntax in Make.com: in a filter’s left field, wrap the field reference: lower({{1.Platform}}) — normalises the input before comparison regardless of how the team typed it.

Build a live social scheduling automation: Google Sheets → Make.com Router → Buffer. Follow every step exactly. Do not move forward until the current step is verified.

Step 1
Set Up Google Sheets
  1. Go to sheets.google.com → create a new spreadsheet → name it Content Calendar.
  2. In Row 1 add these exact column headers, one per cell: Platform  |  Caption  |  Hashtags  |  Post Date  |  Status
  3. Add 3 test rows (rows 2–4). Use real post text, not placeholder text:
    • Row 2: instagram | your caption | #hashtag1 #hashtag2 | today’s date | (leave blank)
    • Row 3: twitter | your caption | (leave blank) | today’s date | (leave blank)
    • Row 4: linkedin | your caption | (leave blank) | today’s date | (leave blank)
  4. Critical: Platform values must be lowercase, no spaces. Make.com reads this cell exactly as typed.
Step 2
Connect Buffer Channels
  1. Go to buffer.com → log in or create a free account.
  2. In the left sidebar click “Connect a channel” (or “Add a channel”).
  3. Select a platform → complete the OAuth flow. Connect at least one Instagram Business, Twitter/X, and LinkedIn account. Buffer’s free plan supports 3 channels.
  4. Once connected, each channel appears in your Buffer dashboard. Note the exact channel name — you will select it inside Make.com later.
  5. Verify: Buffer’s queue should show 3 connected channels with empty queues, ready to receive posts.
Step 3
Build the Make.com Scenario

A — Create the scenario

  1. Go to make.com → log in → click the blue “Create a new scenario” button (top right of dashboard).
  2. The scenario editor opens with an empty canvas and a large “+” in the center.

B — Add the Google Sheets trigger

  1. Click the center “+” → in the search bar type Google Sheets → select Google Sheets → then select the module “Watch Rows”.
  2. Click “Add” next to the Google connection → a popup opens → log in with your Google account → click “Allow” on both permission prompts.
  3. Back in the module settings panel, fill in:
    • Spreadsheet ID: click the field → your spreadsheets load → select Content Calendar
    • Sheet Name: select Sheet1 (the default tab)
    • Table contains headers: set to Yes
    • Row with headers: enter 1
    • Where to Start: click “Choose manually” → in the dialog select “All” → confirm
  4. Click OK. A Google Sheets module icon now appears on the canvas.

C — Add the Router

  1. Hover over the Sheets module → a small “+” appears on its right edge → click it.
  2. Search Router → select “Router” (it is a Make.com built-in, not a third-party app).
  3. The Router appears connected. It starts with one route. Click “Add route” twice → you now have 3 routes.

D — Instagram route (Route 1)

  1. Click the wrench/filter icon on Route 1 (the small icon on the line between the Router and the empty circle).
  2. In the filter dialog:
    • Label: type Instagram
    • Left field (condition): click the field → look for the formula toggle (labelled f(x) or a formula mode button) → type lower({{1.Platform}}) — the {{1.Platform}} references the Platform column from Module 1 (your Sheets trigger)
    • Operator: select “Text operator” → “Equal to”
    • Right field: type exactly instagram (all lowercase)
  3. Click OK.
  4. Click the “+” at the end of Route 1 → search Buffer → select Buffer → select module “Create a Status Update”.
  5. Connect your Buffer account if prompted (OAuth popup).
  6. In the Buffer module settings:
    • Profile ID: click the dropdown → select your Instagram channel
    • Text: click the field → from the Sheets module output, click Caption to insert it. Then click after it in the field and also insert Hashtags (final text = Caption + space + Hashtags)
  7. Click OK.

E — Twitter/X route (Route 2)

  1. Click the filter icon on Route 2 → Label: Twitter → left field formula: lower({{1.Platform}}) → Equal to → right field: twitter → OK.
  2. Add Buffer — Create a Status Update → Profile ID: your Twitter/X channel → Text: Caption only (no hashtags for Twitter) → OK.

F — LinkedIn route (Route 3)

  1. Click the filter icon on Route 3 → Label: LinkedIn → left field: lower({{1.Platform}}) → Equal to → right field: linkedin → OK.
  2. Add Buffer — Create a Status Update → Profile ID: your LinkedIn channel → Text: Caption only → OK.

G — Fallback route (Route 4)

  1. Click “Add route” on the Router → Route 4 appears. Do NOT add a filter — leave it empty. Make.com treats a filterless route as the fallback for everything that didn’t match routes 1–3.
  2. Add Google Sheets — Update a Row:
    • Spreadsheet ID: Content Calendar
    • Sheet Name: Sheet1
    • Row Number: click the field → from the Sheets module (Module 1) output, select “Row Number” (the system field that identifies which row triggered this run)
    • Values → Column E (Status): type FAILED - unknown platform
  3. Click OK.

H — Add “Scheduled” status updates to success routes

  1. On each of Routes 1, 2, and 3 — click the “+” after the Buffer module → add Google Sheets — Update a Row with the same settings as Route 4, but Status column value = Scheduled.

I — Save and name

  1. Click the scenario title (top left, shows “New scenario”) → rename to Content Calendar → Buffer.
  2. Click the blue Save button (top right).
Step 4
Test and Verify
  1. In Make.com, click “Run once” (bottom left of the scenario editor). The scenario runs against your 3 test rows.
  2. Open Buffer → confirm 3 posts appear in the correct channel queues (Instagram, Twitter/X, LinkedIn).
  3. Open your Google Sheet → confirm the Status column (E) shows Scheduled for rows 2, 3, and 4.
  4. Add a 4th test row: Platform = Pinterest, Caption = any text, leave the rest blank.
  5. Click “Run once” again → open the sheet → row 5 Status should show FAILED - unknown platform.
  6. In Make.com, click History (top menu) → all runs should show a green checkmark. If any show orange or red, click the run to see which module failed.

Mark complete only when: Buffer shows 3 scheduled posts AND the sheet shows “Scheduled” for rows 2–4 AND the Pinterest row shows the FAILED message.

Troubleshooting
  • lower() not working / can’t type it: you must be in formula mode. Click the f(x) icon or formula toggle in the filter left field. Then type the full expression: lower({{1.Platform}}). The 1 refers to Module 1 (your Sheets trigger).
  • Buffer says “no profile found”: the Buffer connection expired. Re-open the Buffer module → click the Profile ID field → if channels don’t appear, click “Add” to reconnect via OAuth.
  • Status column not updating: verify the Update Row module uses Row Number from Module 1’s output — not a hardcoded number and not a value from another module.
  • Posts going to the wrong Buffer channel: re-check the Profile ID dropdown in each Buffer module. Make.com can auto-select the wrong profile if you have multiple Buffer accounts connected.
  • Free tier math: Make.com free = 1,000 ops/month. Your current Scenario has ~5 modules. 5 modules × 4 test rows = 20 ops. At 200 posts/month production: 5 × 200 = 1,000 ops — you hit the ceiling. Upgrade before a client goes live at scale.
Day 9 · Session 8

Common Mistakes to Avoid

These are the most frequent errors from VAs building their first conditional automations — and from applying this week's lessons in real client work.

Automation Mistakes

  1. No fallback route. The Scenario fails silently when an unexpected value appears. You find out a week later when someone asks why a post was never published.
  2. Case sensitivity ignored. "Instagram" and "instagram" are different strings. Use the lower() text function in your filters to normalize text before routing.
  3. Testing only the happy path. You test Instagram, it works, you turn it on. You never test Twitter, LinkedIn, or the fallback. One month later, a LinkedIn row breaks the Scenario and nobody notices for two weeks.
  4. No log. When something goes wrong, you have no record of what triggered, when, or what the outcome was. Debugging becomes a guessing game that wastes hours.
  5. Adding complexity too early. A 4-route Scenario with 3 modules each is not a starter project. Build one route, test it completely, then add the next. Complexity multiplies problems — especially untested ones.

General Mistakes from This Week

  1. Trusting AI output without verification. AI answers confidently — even when wrong. Every deliverable that goes to a client needs a human check first.
  2. Using the wrong tool because it is familiar. Defaulting to one AI for everything instead of picking the right tool for the task. Familiarity is not the same as fit.
  3. Skipping context in prompts. "Write a LinkedIn post" and "Write a 150-word LinkedIn post for a staffing agency CEO targeting HR managers, professional tone, ends with a question" are not the same prompt. One produces a generic result. The other produces something usable.

Social Media Scheduling Automation

A real automation you can build today using only free tools you already have. No API keys, no paid upgrades — just Google Drive, Zapier (free tier), and the GHL Social Planner.

The Pipeline

Google Drive
Trigger: new file added to a “Ready to Post” folder
Zapier
Reads the file + caption, formats it for GHL
GHL Social Planner
Post is scheduled to the client’s connected social accounts

How It Works — Step by Step

  1. Set up a Google Drive folder — name it something like Ready to Post. This is where the VA drops the creative (image or video) and a short text file with the caption and hashtags.
  2. Create a Zapier Zap (free tier) — Trigger: “New File in Google Drive Folder.” Zapier watches your folder and fires every time a new file lands.
  3. Add a Zapier action: GoHighLevel — Zapier has a native GHL integration (OAuth — no API key needed). Use the “Create Social Planner Post” action. Map the image URL and caption from Drive into the GHL fields.
  4. GHL Social Planner publishes — the post goes into the client’s social queue. GHL handles the scheduling and publishing to connected platforms (Facebook, Instagram, Google Business Profile).

Why This Is Free

What You Use

  • Google Drive — free with any Google account
  • Zapier Free Tier — 100 tasks/month, 5 Zaps. Plenty for a demo or low-volume client
  • GHL Social Planner — already included in every GHL plan. No extra API calls

What You Skip

  • No paid Zapier plan needed at this volume
  • No GHL API key or developer account
  • No third-party scheduling tool (Buffer, Hootsuite, etc.)
  • No custom code or webhooks
Zapier Free Tier Limits

100 tasks/month means roughly 100 posts. If a client posts once per day per platform, that’s ~30 tasks/month for one client — well within the free limit. If you scale past that, Zapier’s Starter plan or switching to Make.com (which has a more generous free tier) are the next moves.
