Day 3 · Session 2

Live Artifacts

A regular artifact is built once and frozen, like a document. A live artifact is different: it connects to your real tools and pulls fresh information every time you open it. Think of it as a personal dashboard that stays up to date on its own.

Regular artifact
Built once. Claude makes it from your prompt and the content is frozen at that moment. Great for a doc, email, or table you need right now. Copy it, use it, done.
Live artifact
Always fresh. Claude builds it and connects it to your tools. Every time you open it, it pulls the latest: new emails, new Slack messages, today's calendar. A page you come back to.
What it connects to

If you have connected a tool in Cowork, a live artifact can read from it with no extra setup: Slack (messages, channels, threads), Gmail (emails, drafts), Calendar (meetings, schedules), and Files (documents, folders).

What VAs actually build

Live artifacts shine for anything you check repeatedly. Instead of digging through five tabs, you open one page that refreshes itself.

Live dashboardWhat it shows
Morning briefingToday's calendar, priority emails, and Slack mentions on one page. Open it each morning, no typing needed.
Client email trackerEvery email from key clients in the last 7 days, so you see what needs a reply without searching your inbox.
Team update boardThe last 24 hours of Slack across your channels: what is urgent, what is waiting on you, what you can skip.
The Principle

Regular artifact = something you build once. Live artifact = something you open every day. If you keep checking the same thing, make it a live artifact.

MCP & API

These are the behind-the-scenes pieces that let Claude talk to Slack, read your Gmail, and check your calendar. You will never touch them directly, but knowing what they are helps you understand what is happening when Claude connects to your tools.

API = the menu
Every app (Slack, Gmail, Calendar) has a set list of things it lets other software do: read messages, search emails, list events. That list of allowed actions is the API. Like a restaurant menu: it says what Claude can order and how to ask.
MCP = the translator
Every app's menu is written in a slightly different language. MCP is the universal translator that lets Claude read all of them the same way, instead of learning each app separately.
Think of it this way

You walk into a restaurant abroad. The menu is in a language you do not speak. MCP is the translator who reads it and places your order. The menu itself, the dishes and how to customize them, is the API.

When you ask Claude to check your Slack

This whole chain happens in under a second. You type in plain English and the data comes back.

You + Claude
ask in plain English
MCP
the translator
Slack (API)
the app's menu
request →
← data
Two legs, both automatic

Green, going out: your plain-English request runs through MCP, which translates it into the exact call the app's API expects. Gold, coming back: the app returns the data, MCP hands it to Claude, and you see the answer. The whole round trip takes under a second.

Why you should know this

ReasonWhy it helps you
TroubleshootingWhen something fails, knowing the chain helps you find where it broke: the app, the connection, or the way you asked.
Talking to clientsClients ask "how does this work?" Explaining it simply builds trust and shows you know your tools.
Choosing toolsWhen a client wants to connect a new app, you will know whether it is possible and how it fits.
The simple version

You never have to think about API or MCP. You ask Claude what you need in plain English; the rest is automatic. Knowing the plumbing just makes you better at the job.

What Is a CLI?

CLI stands for Command Line Interface. It is a way to use a computer by typing commands instead of clicking buttons. You type what you want, hit Enter, and it happens. No menus, no icons, just text.

Buttons and clicking
What you are used to. Chrome, Gmail, Slack, Canva: anything with buttons, menus, and a mouse. This is a GUI, a graphical interface. You see things and click them. Easy and visual.
Typing commands
What a CLI looks like. A plain text window where you type instructions. No buttons. It looks like a hacker movie, but it is just a faster way to talk to your computer. It is how developers and many AI tools work.
Think of it this way

Clicking buttons is like ordering food by pointing at pictures: anyone can do it. Typing commands is like telling the chef exactly what you want by name: faster and more precise, but you need to know the words.

A real example

Here is a CLI in action. The green text is waiting for your input. You type a command, hit Enter, and get a result. That is the whole loop.

Terminal

$ ls Documents/Clients/

# Find a name inside a file
$ grep "Scarlett" MOCK_DATA.csv

# Count how many contacts are in the file
$ wc -l MOCK_DATA.csv
39 MOCK_DATA.csv

Cowork or Claude Code?

Same Claude underneath, two different doors. Pick the one that matches the job.

Cowork (most VAs)
A friendly desktop app. You point, talk, and watch. Connected to your everyday tools. Best for daily work: drafting, organizing, and running tasks across apps.
Claude Code (builders)

The Principle

Start in Cowork. Reach for Claude Code when the work turns into building or automating at scale. Same brain, two doors.

Organize and Extend Claude

Three features turn Claude from a general assistant into your specialist. Projects keep your context, Skills teach repeatable tasks, and Plugins bundle it all so a whole capability installs in one click.

FeatureWhat it isUse it to
ProjectsA saved workspace with your files and custom instructionsKeep recurring work in context: a client, a campaign, an account
SkillsA small folder (a SKILL file) that teaches Claude one repeatable taskMake Claude do a task your way, the same way, every time
PluginsA bundle of skills and tools you install at onceAdd a whole capability set in one click, and share it with the team
Plain version

A Project remembers your stuff. A Skill teaches a trick. A Plugin is a box of tricks you can hand to anyone.

How they work together

They stack. Here is one job, built up across all three.

Example: client onboarding

Project: an "Acme Onboarding" workspace holds Acme's docs, brand voice, and instructions.

Skill: a "welcome sequence" skill writes the onboarding emails in your exact format.

Plugin: an "onboarding pack" bundles several skills so every VA on the team installs them at once.

The Principle

Projects remember, Skills specialize, Plugins distribute. Together they turn a generalist into your team's specialist.

What a Project Remembers

A Project is not just a folder. It holds everything Claude needs before you type a single message -- the client, the rules, the tone, the files. That context is ready the moment you open it.

Regular chat
Starts blank every time. Claude does not know your client, your tone, or what you discussed last week. You re-explain from scratch.
Inside a Project
Opens with memory. Claude already knows the client brief, the rules, the brand voice. Your first message is the task, not the introduction.
What you storeWhat Claude gains
System promptStanding role and rules: who Claude is, what tone to use, what never to do -- set once, active always
Files and docsBrand guides, SOPs, previous reports -- Claude reads them before you ask anything
Chat historyPast conversations in this Project stay visible, so nothing gets forgotten between sessions
The Principle

One Project per client or recurring responsibility. Everything Claude needs to know about them lives there. You stop re-explaining -- Claude already knows.

Day 3 · Session 7

Context Engineering

Prompting is what you ask. Context engineering is everything you put around the ask: the system prompt, the files, the tools, and the memory. It is the difference between a generic answer and one built for your exact job.

First, the system prompt

A system prompt is the standing instruction that sets who Claude is and the rules it follows for a whole session or Project, before you type a single message. "You are a careful bookkeeping assistant. Never invent numbers." That one line shapes everything after it.

Thin context
Just a question, no setup. "Write the client update." Claude has to guess the client, the tone, and the facts, and usually guesses wrong.
Engineered context
The system prompt sets the role and rules, the client's last report is attached, and your tone example is included. The same ask now produces drop-in work.

Why context is the real skill

The model can only be as good as what you put in front of it. A brilliant model with thin context gives thin answers. An average model with rich, relevant context gives great ones.

Put in the contextKeep it out
The role and rules (system prompt)Irrelevant chat history
The exact source material it needsFiles it does not need this time
One or two examples of "good"Conflicting or outdated instructions
The tools or connectors for the taskEverything "just in case" (it crowds the window)
The Principle

The best operators do not write cleverer one-liners. They engineer better context: the right role, the right files, the right examples, and nothing that distracts.

Claude Secrets

Habits That Separate Power Users

Most people use Claude like a search engine. These six habits change that. Each one takes seconds to apply and noticeably improves what you get back.

01
Ask Claude to critique itself

"What is wrong with that answer?" triggers a sharper second draft. Claude catches its own weak points better than most editors.

02
Upload the actual file

Do not paste text from a PDF -- upload the file. Claude reads the full document including tables and layout. Pasting loses that.

03
"Think step by step"

Add this before a complex task. It forces Claude to reason before concluding -- dramatically fewer wrong answers and invented facts.

04
Set your system prompt once

A Project system prompt shapes every response. "You are a concise EA. Never use bullet points." -- set it once, active in every session.

05
"One more thing"

After Claude finishes any answer, type exactly this. It reliably surfaces the most important insight Claude left out the first time.

06
End with "plain English"

Add "explain the key point in one plain sentence" to any prompt. Forces Claude to name what actually matters instead of padding.

What this unlocks

Autopilot, with a seatbelt

Unattended work is powerful, which is exactly why the Day 1 rules still apply, even more so when no one is watching.

Same rules, higher stakes

Anything that runs on its own needs the same data-safety guardrails and a human checkpoint for risky steps. Schedule the gathering and drafting; keep a person on the send and approve.
