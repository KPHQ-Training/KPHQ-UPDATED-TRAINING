1. What is n8n and why does it matter for your role?

The short version: it does repetitive tasks automatically, so you don't have to

n8n (pronounced "n-eight-n") is a tool that connects apps and automates tasks. Think of it as a set of digital instructions you build once, and then it runs on its own every time something happens — a new lead fills out a form, a Google Sheet row gets updated, a date arrives.

As an EA for a real estate agent, you deal with:

  • New leads coming in from Zillow, the agent's website, or a form
  • Buyers and sellers at different stages of a transaction
  • Tasks that must happen at the right time (inspection deadlines, closing reminders)
  • Routine emails and follow-ups that feel the same every time

Without automation, you handle all of this manually. With n8n, you build the rule once and the system handles it automatically every time it happens — even when you are not at your computer.

Real-world analogy

Imagine you told a very reliable assistant: "Every time a new buyer form comes in, if their budget is over $500K, text the agent immediately. If it's under $500K, add them to the follow-up email list." n8n is that assistant — except it never forgets, never sleeps, and handles dozens of leads at once.

n8n Cloud vs n8n self-hosted

n8n has two versions. n8n Cloud is the subscription version you access at app.n8n.cloud — it runs on the internet, you log in from any browser, and nothing needs to be installed. Self-hosted means running n8n on your own computer or server, which requires technical setup.

As a real estate EA, you will use n8n Cloud. Everything in this guide is written for that version. You do not need to install anything.

2. Key terms you need to know

Learn these six words and the rest of this guide will make sense
Workflow
The full automation from start to finish. A workflow is the entire sequence of steps — what starts it, what it checks, what it does. Think of it as one complete process, like "Handle new buyer lead."
Node
A single step inside a workflow. Each box you see on the canvas is a node. One node might fetch data from a Google Sheet. Another might send a Gmail. Another might check a condition. Nodes connect to each other like a chain.
Trigger
The first node in any workflow — the thing that starts everything. A trigger is always an event: "A form was submitted," "A Google Sheet row was changed," "It's 9am on Monday." Nothing runs until the trigger fires.
Action
A node that does something: sends an email, posts a Slack message, creates a task, adds a row to a spreadsheet. Actions are the "results" of your workflow.
Condition
A yes/no question the workflow asks before deciding what to do next. "Is this buyer pre-approved? Yes or no." The answer determines which path the workflow takes.
Canvas
The visual workspace in n8n where you build your workflow. It looks like a blank whiteboard with nodes connected by lines. You drag, drop, and connect nodes here.
💡
Read this first

Every workflow in n8n follows the same pattern: Trigger → (optional condition) → Action. Everything you build — no matter how complex it looks — is just a combination of these three things.

4. What is a condition?

The "only if" rule that makes your automation smart

Without a condition, every workflow runs the same steps every time. It does not matter who the lead is, what stage the deal is at, or how urgent the situation is — the same email goes out, the same task gets created.

A condition adds judgment. It pauses the workflow, asks a yes/no question about the data, and then takes a different path depending on the answer.

Without a condition
New lead submits buyer form → Send one generic follow-up email.

A $1.2M cash buyer ready to close in 30 days gets the same email as someone who clicked an ad with no budget and no pre-approval. The agent never knows to prioritize.
With a condition (IF node)
New lead submits buyer form → Is budget over $500K AND pre-approved?
Yes: agent gets an immediate Slack alert + a "Call within 1 hour" task is created.
No: lead enters a 5-day automated email sequence.

The agent knows immediately who needs their attention.
Think of it like a phone tree

You have used automated phone systems that say "Press 1 for sales, press 2 for support." That is a condition — depending on what you press (the data), the call goes somewhere different. n8n conditions work the same way, but with your real estate data instead of key presses.

Every condition has three parts

PartWhat it isReal estate example
The fieldThe piece of data you are checkingThe buyer's budget amount
The operatorHow you are comparing it"is greater than"
The valueWhat you are comparing it against500000

Together they form one rule: Budget is greater than 500000. That is a complete condition.

5. The IF node: two-path decisions

One condition, two possible routes — true and false
IF node — found under Flow in the node picker

The IF node is n8n's main conditional tool. You give it one question to ask. Based on the answer, it sends the data one of two ways:

  • True output — the condition was met. The data goes this way.
  • False output — the condition was not met. The data goes this way instead.

The big difference from other tools you may have heard of (like Zapier's free Filter): in n8n, both the true and false outputs are usable. You can connect a different action to each one. You are not forced to just "stop" when the condition fails.

What it looks like on the canvas

The IF node appears as a box with two output dots on the right side — one labeled true (top) and one labeled false (bottom). You drag a separate wire from each output to a different next step.

↓

6. Building your first IF node (step by step)

Literal instructions — exactly what to click and where
Before you start — run the trigger first

When you open the IF node settings, you need to map a field from your trigger. But n8n can only show you those fields if it has seen real data first. Before touching the IF node, click Test workflow and submit one test entry through your trigger (fill out the form, update a row in the sheet, etc.).

Once you do that, the data from your test appears inside the IF node and you can click to select fields instead of typing them manually. Skip this step and you will be working blind.

1
Click the + button after your trigger node
On the right edge of your trigger node (the first box on the canvas), click the + button. The node picker panel opens on the right side of the screen.
2
Search for "IF" and select it
In the search box at the top of the node picker, type IF. You will see "If" appear under the Flow category. Click it. The IF node appears on the canvas connected to your trigger.
You will see the IF node has two output dots on the right side — true (top) and false (bottom).
3
Double-click the IF node to open its settings
A settings panel opens on the right side. You will see a section called Conditions with an Add condition button.
4
Click "Add condition" and set the data type
Click Add condition. A row appears with a dropdown on the left. This dropdown sets the data type — what kind of value you are checking.
String = text values. Use for: lead type ("Buyer"/"Seller"), status ("Approved"), lead source ("Zillow").

Number = numeric values. Use for: budget ($500,000), days on market (30), price per sq ft.

Boolean = true/false values. Use for: pre-approved (yes/no), contract signed (yes/no).

⚠️ This matters a lot. If you set a price field as "String" and compare it as a number, the comparison will be wrong and leads will silently route to the wrong path. Set this correctly before anything else.
5
Map Value 1 to the field you want to check
Click inside the Value 1 box. You will see a small {"{{"} icon appear. Click that icon to open the data picker. Your trigger's fields appear as a list — click the field you want to check (e.g. "budget" or "lead_type"). n8n fills in the expression automatically.
If the data picker shows no fields: close the IF node, click Test workflow, submit a test entry, then come back and try again. The fields only appear after n8n has seen real data.
6
Choose the operator
The middle dropdown is the operator — how you are comparing the field. Click it and choose the rule that fits your situation. Common ones for real estate:
is equal to — the field must exactly match your value. Use for status fields ("Approved"), lead type ("Buyer"), lead source ("Zillow").

is greater than — the number must be above your value. Use for budget, price, days on market.

contains — the text includes your value anywhere. Use for notes fields (notes contains "cash offer"), email addresses, or any field where exact match is too strict.

is not empty — the field has any value at all. Use to check if a phone number was provided before sending a text.
7
Type the comparison value in Value 2
Click Value 2 and type what you are comparing against. For a budget check: 500000 (no dollar sign, no comma — just the number). For a status check: Approved (exact capitalization matters for "is equal to").
⚠️ Case sensitivity warning: "is equal to" is case-sensitive for String fields. "approved" will not match "Approved." Match the capitalization exactly as it appears in your form or sheet.
8
Connect the true output to your first action
Close the settings panel. On the IF node, find the output dot labeled true (top right). Click the + next to it, search for your action node (e.g. Slack, Gmail), and add it. This is what happens when the condition passes.
9
Connect the false output to your second action
Find the output dot labeled false (bottom right). Click its + and add the action for when the condition fails. If you want nothing to happen when the condition fails, you can leave the false output unwired — the data simply stops there.
10
Test with both a passing and a failing value
Click Test workflow. Submit one entry where the condition should pass (budget = 750000) and one where it should fail (budget = 200000). Watch the canvas — n8n highlights the path each item took. Confirm each went to the right output before activating the workflow.

7. Multiple conditions: AND vs OR

When one rule is not enough to make a decision

Sometimes one condition is not specific enough. You might not want to alert the agent for just any high-budget lead — you want to alert them for a high-budget lead who is also pre-approved and ready to move. For that, you add a second condition and combine them with AND.

Inside the IF node settings, after you add a second condition, a toggle appears at the top: AND or OR.

AND — every condition must pass
Pre-approved = "Yes"
AND Budget > 400,000

Both must be true for the item to go to the true output. If either one fails, it goes to false. Use AND when a lead must meet all your criteria together.
OR — any one condition is enough
Lead source = "Referral"
OR Notes contains "cash"

If either is true, the item goes to the true output. Use OR when any one of several signals should trigger the same action.

How to add AND/OR conditions in n8n

1
Open the IF node settings and add your first condition
Follow the same steps from Section 6. Set your first condition completely before adding a second one.
2
Click "Add condition" again
A second condition row appears below the first. A toggle labeled AND or OR appears between them (or at the top of the conditions block). It defaults to AND.
3
Set up the second condition exactly like the first
Choose the data type, map Value 1, choose the operator, and enter Value 2 for your second rule.
4
Choose AND or OR using the toggle
Click the toggle to switch between AND and OR. AND = all conditions must pass. OR = any one condition is enough. Pick the one that matches your decision logic.
Real estate example: AND in action

You only want to create a priority task for buyers who are both pre-approved and have a budget over $400K. A buyer who is pre-approved for $200K is not a priority lead. A buyer with a $700K budget who is not pre-approved is not ready yet.

Condition 1 (String): pre_approved is equal to "Yes"
AND
Condition 2 (Number): budget is greater than 400000

8. The Switch node: three or more paths

When yes/no is not enough — for stages, lead sources, and client types
Switch node — found under Flow in the node picker

The IF node is perfect for two outcomes. But real estate has situations with three, four, or more distinct outcomes from a single field. A transaction moves through multiple stages. Leads come from multiple sources. Each one might need a different action.

That is what the Switch node is for. Instead of two outputs, it has one output per rule you add — plus a fallback for anything that does not match.

Think of it like a post office sorter

A postal sorter reads the zip code on each envelope and sends it to a different bin: Bin 1 for 90210, Bin 2 for 10001, Bin 3 for 60601. Each envelope goes exactly where it belongs. Anything with an unrecognized zip code goes to a "miscellaneous" bin — that is your fallback output. The Switch node works the same way with your data.

Example: Transaction stage routing

Every time a row in your Transaction Tracker sheet is updated, the workflow checks the Stage field and triggers the right action for that stage automatically:

Trigger
Google Sheets: Transaction Tracker row updated
Fires whenever any row in the sheet changes
↓
Switch Node — checks the Stage field
What is the value of "Stage"?
Routes to Output 1, 2, 3, 4, or fallback based on the value
4 outputs below ↓
Output 1 — "Under Contract"
Create inspection task (7-day deadline) + email TC
Output 2 — "Inspection Done"
Create appraisal task + Slack: #transactions
Output 3 — "Clear to Close"
Send closing checklist email to buyer/seller
Output 4 — "Closed"
Send thank-you + Google review request
The fallback output — do not skip this

The Switch node has one more output at the bottom: the fallback. This catches any item whose value does not match any of your rules. An agent types "under contract" (lowercase) when your rule expects "Under Contract." A new team member types "Pending" when your workflow expects "Under Contract."

Without a fallback, that row is silently ignored. No task gets created. No email goes out. A closing deadline gets missed. Always wire the fallback to a Slack message or a Google Sheet log so you can see when something unexpected came through.

9. Building your first Switch node (step by step)

Literal instructions for the transaction stage example
1
Run the trigger first to load real data
Same rule as the IF node — click Test workflow and update a real row in your Transaction Tracker sheet before opening the Switch node. This loads your field names so you can click instead of type.
2
Click + after your trigger and search "Switch"
Click the + button on your trigger node → search Switch in the node picker → select Switch under the Flow category. The Switch node appears with one rule slot and a fallback output.
3
Open the Switch node and map Value 1
Double-click the Switch node to open settings. In the Value 1 field, click the {"{{"} icon and select the field you are routing on — for this example, select stage from your Google Sheets data. This is the single field all your rules will check.
4
Set up Rule 1
In the first rule row, set the operator to is equal to and type the first stage value in Value 2: Under Contract. This creates Output 1.
⚠️ Type the value exactly as it appears in the sheet — same capitalization, same spacing. "Under Contract" and "under contract" are different values.
5
Click "Add routing rule" for each additional stage
Click Add routing rule and repeat step 4 for each stage: Inspection Done, Clear to Close, Closed. Each rule adds a numbered output connector to the node. You will see Output 1, 2, 3, 4 appear on the right side of the node.
6
Enable the fallback output
Scroll to the bottom of the Switch node settings. Find the Fallback output toggle and turn it on. A fallback output connector appears at the bottom of the node. Do not leave it there unwired.
7
Wire each output to its action node
Close the settings. Click the + next to Output 1 and add the "Under Contract" action (create inspection task + email TC). Repeat for each output. For the fallback, add a Slack message node that sends you an alert with the unrecognized value so you can investigate.
8
Test each route separately
Update your test sheet row to each stage value one at a time and run the workflow. Confirm each value routes to the correct output. Then test an unrecognized value (e.g. type "Pending") and confirm the fallback fires. Only activate the workflow after all routes are verified.

10. What are expressions?

The way n8n reads data from a previous step — you do not need to write these by hand

When you click the {"{{"} icon in the Value 1 field and select a field from the picker, n8n inserts something that looks like this:

Example expression (n8n inserts this automatically)
{{ $json["budget"] }}

This is called an expression. It tells n8n: "take the value of the 'budget' field from the data that just came in." You do not need to write or understand this — just use the data picker and n8n writes it for you.

Reading an expression if you have to

If you ever need to read or understand an expression:

  • $json means "the data from the previous node"
  • ["budget"] means "the field named budget"
  • So {"{{ $json[\"budget\"] }}"} means "the budget value from the previous step"
The most important thing to know about expressions

Always use the data picker (the {{ icon) instead of typing field names by hand. Field names are case-sensitive in n8n. "Budget" and "budget" are different. "Lead Source" and "lead_source" are different. When you click to select, n8n uses the exact correct name. When you type, you risk a typo that silently breaks the condition.

If the picker shows no fields: the trigger has not been tested yet. Run it first.

12. Common mistakes and how to avoid them

Things that trip up beginners — and exactly how to fix each one
Mistake 1 — Setting a price or number field as "String" type

What happens: Your IF node checks budget "is greater than 500000" but uses String type. n8n compares text alphabetically. "$1,200,000" starts with "$" or "1" — both come before "5" alphabetically — so a million-dollar buyer silently routes to the false output. You never know.

The fix: Always set the data type to Number for any field that holds a dollar amount, a count, or a date offset. Check the type dropdown before you do anything else when building an IF node.

How to spot it: Test with a value you know should pass (budget = 750000) and one that should obviously pass but is large (budget = 1500000). If the large one routes to false, your type is wrong.

Mistake 2 — Not testing the false/fallback path

What happens: You test with one passing value, see it work, and assume the other path is fine. The false path was wired to the wrong node, or the fallback was not wired at all. A lead enters the wrong sequence for weeks and nobody knows.

The fix: Always test with at least two values: one that should pass, one that should fail. For Switch nodes, test every single rule AND an unrecognized value to confirm the fallback fires. Mark this as a required step in your workflow QA process.

Mistake 3 — Case sensitivity with text fields

What happens: Your rule says Stage "is equal to" "Under Contract". An agent types "under contract" or "UNDER CONTRACT". The condition fails silently. The transaction falls out of the automation and no tasks get created.

The fix: Either (a) standardize your sheet/form so values always use the same capitalization (use a dropdown instead of freetext), or (b) add a Set node before the IF/Switch node that converts the field to lowercase, and then use all lowercase values in your rules. Option (a) is simpler.

Mistake 4 — Skipping the fallback on the Switch node

What happens: You add rules for four stages but leave the fallback output disconnected. A new team member adds a "Contingent" stage that was not in your rules. Every "Contingent" row is silently dropped. Deadlines are missed.

The fix: Always enable and wire the fallback output. Even if you are confident you have covered all cases, wire it to a "log to Google Sheet" node or a Slack alert that says: "Unrecognized stage value: [value]. Please check." It costs 30 seconds to set up and saves hours of confusion.

Mistake 5 — Budget field arriving as "$500,000" (with symbols)

What happens: Your form sends the budget as "$500,000" with a dollar sign and comma. n8n receives it as text, not a number. Even if you set the type to Number, the comparison breaks because n8n cannot convert "$500,000" to a number automatically.

The fix: Add a Set node between the trigger and the IF node. In the Set node, create a new field called "budget_clean" with the expression {"{{ $json[\"budget\"].replace(/[$,]/g, '') }}"}. This strips the dollar sign and commas, leaving just "500000". Then use budget_clean in your IF condition instead of budget.

Mistake 6 — Activating the workflow without testing it

What happens: You build the workflow, click Activate, and live leads start flowing through. A misconfigured condition routes 20 high-value leads to the wrong sequence. You only notice days later.

The fix: Never activate a workflow until you have tested every branch with real (or realistic) test data. Use n8n's Test workflow button to run the full workflow manually before turning it on. When you are satisfied every path works correctly, then click Activate.
