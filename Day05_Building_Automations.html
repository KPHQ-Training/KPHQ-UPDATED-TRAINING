Three steps before you write any code.

Setup guide

Track A · Google Apps Script

Step 2: Write the Trigger Function

Paste this into the Apps Script editor, replacing any existing code. Save with Ctrl+S.

Apps Script — Code.gs
function notifyOnSubmit(e) {
// e.values = array of all form answers in order
var name = e.values[1];
var email = e.values[2];
var message = e.values[3];
MailApp.sendEmail({
to: Session.getActiveUser().getEmail(),
subject: 'New form response from ' + name,
body: 'Name: ' + name + '\n' +
'Email: ' + email + '\n' +
'Message: ' + message
});
}
Connect the trigger

In Apps Script: click the alarm-clock icon (Triggers) on the left sidebar > Add Trigger > Choose function: notifyOnSubmit > Event source: From spreadsheet > Event type: On form submit > Save.

Day 5 · Free Tool Build

Map Your Data: Trigger Fields → Action Fields

Every automation moves data from one place to another. Mapping is telling it exactly which field goes where — get it wrong and the automation runs, but the output is garbage.

Example: form submission → email alert
Trigger field→Action field
Form answer 1 (Name)→Email subject: "New response from [Name]"
Form answer 2 (Email)→Email "Reply-To" address
Form answer 3 (Message)→Email body
Four mapping mistakes that break a build
  • Wrong field selected — mapping "Email" into the field meant for "Name" because two dropdown options looked similar
  • Off-by-one index — in Apps Script, e.values[1] pulls whichever question is first on the form. Reorder your form questions later and every index shifts — re-check the mapping after any form edit
  • Unmapped required field — an action field marked required with nothing mapped to it fails the entire run, not just that field
  • Type mismatch — mapping a date field into a plain-text slot, or a number into an email-address field, produces output that looks fine in testing and breaks on real data
The Check

After you map every field, run one test and read the actual output — not just "did it send," but "does every field contain what it's supposed to."

Before You Go Live: QA Checklist

Run every item before connecting to a client's real data.

7-point QA checklist
  1. Tested on dummy data, not real client data
  2. Verified the trigger fires correctly (check execution log)
  3. Confirmed the output arrived where expected (email, sheet, Slack)
  4. Checked that a failed run sends an alert rather than silently failing
  5. Named the automation clearly (what it does, not when you built it)
  6. Documented the trigger, actions, and any credentials in a shared doc
  7. Got client approval before connecting to live accounts
The Rule

A slow QA now means zero late-night fixes later. Test on dummy data every time, even for a "simple" change.

Day 5 · Make It Smart

What Is a Condition?

Most automations run the same steps every time. A condition adds judgment: check one thing, then decide what happens next.

Without a condition
Form submitted → Email sent. Every submission gets an email — a test entry, a spam entry, a duplicate. All treated the same.
With a condition
Form submitted → Is Name field filled in? → Yes: send email. No: stop. Only real, complete submissions trigger the action.
Trigger
New form submission
↓
Condition / Filter
Is the Status field = "New Lead"?
↙      ↘
YES — Continue
Send email alert
NO — Stop
Do nothing
Important distinction

In Zapier, a Filter only has one path: proceed or stop. In Make.com, a Router can have multiple paths — yes does one thing, no does something else. In Apps Script, an if/else handles both paths in code.

Two Free Ways to Add Logic

Same judgment, different tools. Pick whichever matches the track you built on today.

Zapier · Filter step
  1. Add a Filter between trigger and action
  2. Click "Only continue if…" → pick a field
  3. Choose an operator and value
  4. Test with a passing and a failing example
Apps Script · if/else
if (status === 'Approved') {
  MailApp.sendEmail(...);
} else {
  // do nothing
}
Zapier operatorUse it when
Text exactly matchesField must equal a specific word — case-sensitive
Text containsField includes a keyword anywhere — not case-sensitive
Number greater thanNumeric threshold, e.g. invoice amount > 500
The gotcha that breaks the most filters

"Exactly matches" is case-sensitive. "approved" will not match "Approved" — the automation silently stops. Use Text contains for case-insensitive matching, or standardize your form/sheet values.

Where VAs Use This

Four conditions you will actually build, including the one every automation needs: has this already run?

ScenarioConditionYes pathNo path
New lead formBudget field > $1,000Notify sales teamAdd to nurture list only
Content approvalStatus = "Approved"Post to BufferStop — wait for approval
Invoice follow-upDays overdue > 7Send reminder emailDo nothing
Duplicate preventionIs this record ID already logged in the Sheet?Stop — already processedAdd row, then run the action
The Standard

Every automation that can run twice on the same input needs a duplicate check. It is the same "only if" filter you already know — just asking "has this already been done?" instead of "does this qualify?"

Common mistakes
  • Forgetting to test the NO path — most people test with a passing value and call it done. Always test a value that should fail, and confirm the automation actually stops.
  • Filtering before the data is clean — if a field sometimes holds "Approved," "approved," or "Yes," your filter will miss half the triggers. Fix the input source first.
  • Nested if/else inside if/else inside if/else — if you are three levels deep, you need a lookup table in Sheets, not more logic.
Day 5 · Content & Social

Approved Content That Never Gets Posted

The bottleneck is not creation. It is the handoff from approved to live.

Without a pipeline
Content approved in Slack. Buried by notifications. Re-approved 3 days later. Copy-pasted to wrong platform. Posted late. No record of what went out.
With a pipeline
Content approved in Sheet. Automation fires. Posted to platform at scheduled time. Row updated to "Posted." Log is always accurate.
What this flow replaces

Manual copy-paste, missed posting slots, inconsistent timing, manual status updates, and the anxiety of "did that go out?"

Build the Content Flow

One approved row in your Sheet triggers the entire chain.

Sheet Trigger
New row = "Approved"
Filter
Status = Approved
Post Content
Buffer / Hootsuite
Update Row
Status to "Posted"
Sheet structure that makes this work

Columns: Post copy | Platform | Scheduled date | Status (Draft / Approved / Posted). The automation watches the Status column and acts only on "Approved."

Automate the Distribution, Not the Judgment

The rule

Never automate the approval step. The human reviews the content first, marks it approved, and only then does the automation act. This protects the client's brand and your relationship.

CategoryExamples
Automate freelyScheduling, formatting, cross-posting, logging, reminders, data entry
Keep humanApproval decisions, tone reviews, crisis responses, strategy choices
Gray areaFirst-draft creation (AI is fine), final copy (human must verify before it goes out)

The Real Cost of Manual Content Handoffs

Manual (without pipeline)
  • 45 min/week copy-pasting across platforms
  • Inconsistent posting times
  • Missed slots when you are busy
  • Manual status updates in a separate doc
  • No audit trail if something goes wrong
Automated (with pipeline)
  • 5 min/week: review and mark approved
  • Posts at the exact scheduled time
  • Nothing missed, even on busy days
  • Status auto-updates in the Sheet
  • Full log of every post: what, when, where
The Point

Your value to the client is not copy-paste speed. It is the judgment that decides what gets approved. Automate the rest.

Build It: Step by Step

Your turn
Follow these 5 steps with your chosen tool
  1. Open your tool (Zapier, Make, or Apps Script) and start a new workflow
  2. Set up the trigger: choose the event from your map (form submit, schedule, new row, etc.)
  3. Add the first action: connect the next step and map the data fields
  4. Add any filters or conditions from your "only if" question
  5. Add the final action (send, save, post, log) and turn on the automation
Golden rule for live builds

Build one step. Test one step. Do not add the next step until the current one works. The most common mistake is building five steps and then not knowing which one broke.

VA Standards: Before You Hand It Over

Professional standards checklist
  • Name the automation clearly: what it does, not when you built it ("Weekly Client Report" not "Automation 3")
  • Document the trigger, actions, and any API keys in a shared doc the client can access
  • Test on your own data before connecting to the client's live accounts
  • Tell the client what changed and how to pause or turn it off
  • Schedule a monthly review: check that triggers still fire and outputs still look right
Last check

If it breaks next week, can someone else read your documentation and fix it without you? If not, it is not done yet.
