Conditional Logic
in Automation

Most automations run the same steps every single time. Conditions make them smart — they check whether something is true, then choose what to do next.

What is a condition?

The "only if" gate in your workflow

A condition is a yes/no question the automation asks before deciding what to do next. You have already seen one in the four-question map from Day 4 — it was the "only if" question.

❌ Without a condition
Form submitted → Email sent.

Every single submission gets an email. A test submission from you. A spam entry. A duplicate. All get treated the same.
✓ With a condition
Form submitted → Is Name field filled in? → If yes: send email. If no: stop.

Only real, complete submissions trigger the action.
💡
The concept

A condition does not add steps. It adds judgment. The automation checks one thing and decides: proceed or stop.

How the logic flows

⚠ Important distinction

In Zapier, a Filter only has one path: proceed or stop. In Make.com, a Router can have multiple paths — yes does one thing, no does something else. In Apps Script, an if/else can handle both paths in code.

In Zapier: the Filter step

Free tier · No code · Single condition

Zapier's Filter is the simplest conditional tool. It stops the Zap if your condition is not met. Think of it as a bouncer: only let through what matches your rule.

1
Add a Filter step between your trigger and action
In your Zap editor, click the + between steps → search "Filter" → select "Filter by Zapier"
2
Choose your field
Click "Only continue if…" → select the data field from your trigger (e.g. Status, Email, Name)
3
Set the condition
Pick an operator: Text exactly matches / Contains / Is not empty / Number greater than — then type the value
4
Test it
Zapier shows you "This Zap would have continued" or "This Zap would have stopped." Verify with both a passing and failing example.
⚡ Zapier Paths (paid) vs Filter (free)

Filter (free): proceed or stop. One outcome if the condition fails: nothing happens.

Paths (paid): yes does X, no does Y. Two different actions for two different outcomes. If you need the "no" branch to do something specific, you need Paths — or use Make.com's free Router instead.

In Apps Script: if / else

Free · Full control · Handles both yes and no paths

In code, a condition is written as an if statement. If the condition is true, do one thing. If not, do another.

Apps Script — basic if/else
// When a row is edited in the Sheet
function onEdit(e) {
var status = e.range.getValue();

// Condition: only send email if status = "Approved"
if (status === 'Approved') {
MailApp.sendEmail('you@email.com', 'Row approved', 'A row was just approved.');
} else {
// Condition not met — do nothing (or log it)
console.log('Status is ' + status + '. No email sent.');
}
}

Multiple conditions with else if

When you have more than two outcomes, chain conditions together.

Apps Script — three-way condition
function onEdit(e) {
var status = e.range.getValue();

if (status === 'Approved') {
MailApp.sendEmail('client@email.com', 'Approved!', 'Your request was approved.');

} else if (status === 'Rejected') {
MailApp.sendEmail('client@email.com', 'Not approved', 'Your request was not approved.');

} else {
// Status is anything else — do nothing
}
}
✓ Read it in plain English

Every if statement translates directly: "If status equals Approved, send the approval email. Otherwise if status equals Rejected, send the rejection email. Otherwise, do nothing." Write the plain English first. Then write the code.

Common mistakes

Where VAs get stuck with conditions
✗ Mistake 1 — Case sensitivity

"Exactly matches" in Zapier is case-sensitive. "approved" will not match "Approved." Use Text contains for case-insensitive matching, or standardize your form/sheet values to always use the same capitalization.

✗ Mistake 2 — Forgetting to test the NO path

Most people test the condition with a passing value and call it done. Always test with a value that should FAIL the condition. Make sure the Zap actually stops — don't assume it does.

✗ Mistake 3 — Filtering too early

Putting a filter before you have clean, consistent data in the field you are checking. If users sometimes type "Approved", "approved", "APPROVED", or "Yes", your filter will miss half the triggers. Fix the input source before adding the condition.

✗ Mistake 4 — Nested conditions when a table is better

If you find yourself writing if/else inside if/else inside if/else, stop. You probably need a lookup table in Sheets, not nested logic. Complexity inside an automation is almost always a sign that the data model needs to be cleaner.
